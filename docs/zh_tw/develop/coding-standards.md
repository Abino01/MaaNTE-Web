# 編碼規範

## AI 程式設計規範

### 禁止無腦使用 AI 開發

- 直接向 AI 下達籠統指令，例如「幫我發展 xxx 功能並提 PR」「幫我修這個 bug 並提 PR」。
- 在關鍵模組中，用 AI 產生大量難以維護、難以理解的「黑盒」程式碼。
- 在關鍵模組中提交自己看不懂、無法掌控的程式碼。

> [!CAUTION]
> 禁止在未提供 AI 遊戲介面截圖、介面跳躍邏輯等上下文的情況下，讓 AI 直接編寫 Pipeline。
> MaaFramework 的 Pipeline 強依賴遊戲介面與業務邏輯，缺乏介面資訊的 AI 只能依賴幻覺和專案已有程式碼拼湊，產出程式碼品質極低。
> 充分的資訊至少包括：每個識別節點需提供 `roi` 與模板圖片，並說明介面間的跳轉關係（從哪個介面、點擊什麼、跳到何處）。
> 不符合以上條件的 PR 將直接關閉。

### 推薦的 AI 開發方式

- 先學習本專案的編碼規範，自行做架構設計，或將 AI 的建議作為參考。
- 用 AI 做有目標的增量開發，自行 review 產生程式碼是否符合預期。
- 確認無誤後再提交 PR。

## Pipeline 低程式碼規範

### 命名：PascalCase

節點名稱使用 PascalCase，同一任務內以任務名稱或模組名為前綴。例如 `FishNewEntrance`、`WithdrawMoneyEntrance`、`TouchDetect`。內部/私有節點以 `__` 開頭：`__ScenePrivateWorldEnterBag`。

### 禁止硬延遲

盡量少使用 `pre_delay`、`post_delay`、`timeout`。透過增加中間識別節點避免盲目 sleep。

只在必須等待畫面穩定時使用 `pre_wait_freezes` / `post_wait_freezes`，其他時候應盡量避免延遲。
**不要為了執行穩定而使用延遲，而是透過增加中間節點判斷，因為延遲實際上是在掩蓋問題，在用戶設備存在高延遲時仍然不會穩定。 **

### `next` 第一回合即命中

盡可能擴充 `next` 列表，確保任何遊戲畫面都處於預期中，實現一次截圖就命中目標節點。
**專案一般拒絕一切形式的重試機制，一定要確保一次流程完成所有任務，除非遇到無法解決的問題。 **

### 識別 → 操作 → 再識別

每一步操作都基於識別。

**推薦：** 辨識 A → 點選 A → 辨識 B → 點選 B

**禁止：** 整體識別一次 → 點選 A → 點選 B → 點選 C

例如：

1. 介面跳轉中，需要 辨識跳轉按鈕 → 點選跳轉按鈕 → 辨識介面已跳轉完成。 
_你沒辦法保證點完關閉按鈕之後畫面是否還和之前一樣。極端情況下遊戲彈出新池子公告，直接點下一個節點可能點到抽卡里去。 _
2. 點選會變更帳號資料的按鈕時，需要 識別提交按鈕 → 點選提交按鈕 → 辨識按鈕點選成功。 
_你沒辦法保證每個使用者的網路都是順暢的，點擊按鈕事件未與伺服器成功交互，整個交互介面會卡死不動。 _

### 不要盲目重試、增加限制

**推薦：** 遇到 bug 時找問題根因，詳細到具體哪個節點失敗、哪個辨識不符合預期，去修補對應節點的辨識、操作問題。

**禁止：** 同樣的操作再試一次、盲目地加入 `max_hit`。

1. 點選沒反應時再點一次 → 透過 `pre_wait_freezes`、`post_wait_freezes` 等待畫面靜止，或增加中間節點確認按鈕可點選後再執行。
2. 當某個子任務失敗後再跑一次 → 重試只能略微提高成功率，並不能根本解決問題，只會讓程式碼變得難以維護。
3. 當一個節點出現死循環後加 `max_hit` → 出現死循環一般是識別問題、邏輯缺陷導致，盲目加 `max_hit` 只會讓邏輯中斷。

### 處理彈跳窗和加載

好的流程不是"主線能跑就行"，而是：正常主線能跑、彈窗能處理、加載能等過去、不在目標場景時能自動跳過去。

常見做法是在 `next` 裡掛：

- `[JumpBack]SceneAnyEnterWorld`
- `[JumpBack]SceneClickBlankToExit`
- `[JumpBack]SceneLoading`

### OCR 寫完整文本

`expected` 寫完整文本，不寫半截。多語言處理由 CI 工作流程自動完成。需要片段或手寫正規時加上 `// @i18n-skip` 標記。

### 先重複使用，再新增

寫新節點前，先查已有 Pipeline 是否已有現成能力。優先使用 [場景管理器](./scene-manager.md) 的公共接口，禁止直接引用 `__ScenePrivate*` 內部節點。

## Python CustomAction 規範

Python CustomAction 僅用於處理 Pipeline 難以實現的複雜邏輯。 **流程控制由 Pipeline 負責，Python 只處理困難。 **

一句話：**Pipeline 管流程，Python 管難點。 **

_沒有必要的 Python 邏輯會大幅增加程式碼複雜度，造成下一位開發者開發除錯極為困難。 _

## 配套文件

MaaNTE 裡一個功能改變常常不只改一個地方。

### 新增或修改任務

- `assets/resource/tasks/*.json`
- `assets/resource/base/pipeline/**/*.json`
- `assets/resource/locales/interface/zh_cn.json` 等五種語言文件
- `assets/interface.json`（需註冊 task 檔案）

### 新增 Python CustomAction

- 在 `agent/custom/action/` 下新建 Python 文件
- 在 `agent/custom/action/__init__.py` 中導入並加入 `__all__`

## 資源規範

### 解析度：720p 基準

所有圖片、座標（`roi`、`target`、`box`）均以 **1280×720** 為基準。 MaaFramework 在執行時會根據使用者裝置自動轉換。建議使用 **Maa Pipeline Support** 插件進行截圖和座標換算。

### HDR / 色彩管理

當被提示 "HDR" 或 "自動管理應用程式的顏色" 等功能已開啟時，請勿進行截圖、取色等操作，可能導致範本效果與使用者實際顯示不符。

## 常見坑

| 坑 | 處理 |
|-----|------|
| 模型或依賴目錄缺失 | `git submodule update --init --recursive` |
| 直接引用了 `__ScenePrivate*` 節點 | 應引用 `Interface/Scene/` 目錄暴露的公共介面節點 |
| 只顧主線，不處理彈跳窗/載重 | 把彈窗、載重、中間態視為正常情況 |
| 改了任務但沒補文案 | 五種語言檔案都要加 i18n key |
| OCR `expected` 寫了片段 | 寫完整文本，需要跳過時加 `@i18n-skip` |
| 本地能跑但使用者不行 | 檢查幀率、色彩濾鏡、HDR、解析度是否與基準一致 |
